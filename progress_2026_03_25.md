## 5. Root Cause Analysis — TTT Dead

### Diagnosis Commands
```bash
grep -E "swa:|ema:|lora_ttt" seed42.log
grep -E "ttt|TTT|lora|LoRA|error|Error" seed42.log | head -30
grep -E "swa:applying|ema:applying|Serialized|final_int8" seed42.log
```

### Findings

**Finding 1: TTT completely silent**
Zero TTT-related output in the entire log. Not even an error message. The TTT code was wrapped in execution flow that never reached it OR silently failed.

**Finding 2: Script execution order confirmed**
```
swa:applying averaged 24 checkpoints  ← SWA ran ✓
ema:applying shadow weights decay:0.9999  ← EMA ran ✓
Serialized model int6+zlib: 15703041 bytes  ← Serialized ✓
[NOTHING AFTER THIS]  ← TTT started but process killed/timed out
```

**Finding 3: TTT uses `base_model` not the SWA/EMA model**
Code inspection revealed:
```python
q_val_loss, q_val_bpb = lora_ttt(
    base_model, val_tokens, ...  # ← wrong model, pre-SWA/EMA
)
```

**Finding 4: Per-document LoRA reset is O(n_documents) slow**
TTT iterates over each document in val_tokens, injects LoRA, trains, resets. With 50k documents this takes 10+ minutes — far beyond the eval budget.

**Conclusion:** TTT was architecturally broken:
1. Wrong model used (base_model vs SWA/EMA model)
2. Per-document approach too slow for eval budget
3. Results would never have been logged even if it ran

### Decision
**Do not patch TTT. Rebuild on proven base architecture.**

## 8. TTT Legality Research

### Source Analyzed
`https://samacquaviva.com/projects/ttt-clarification/`

### Three TTT Cases Defined

**Case 1 — ILLEGAL: Train on val, measure on val**
Literally training on test data. Loss can reach 0 with large enough model and epochs. Obviously invalid.

**Case 2 — ILLEGAL: Token-stream TTT across documents**
Adapts on full flattened val token stream. Equivalent to 1 additional training epoch on eval data. Cross-document information leakage. This is what most sub-1.10 submissions are doing.
- Our old TTT implementation was Case 2.
- Most open PRs with scores below 1.10 are using this.
- OpenAI's issue #402 will likely invalidate these when ruled upon.

**Case 3 — LEGAL: Per-document TTT independently**
Split val_tokens at BOS boundaries. For each document: score tokens sequentially, adapt on scored tokens only, move to next token. Never let document J's adaptation affect document K's scoring. Dependency graph is identical to standard autoregressive eval.

**Parallelization note:** Assigning different documents to different DDP ranks is legal — it's just an efficiency optimization since documents are independent.

### Impact on Our Strategy
- Our old TTT: Case 2 (illegal) — would have been rejected
- Target for v3: Case 3 (legal) — implementable, projects ~0.01-0.02 bpb gain
- For now: run clean no-TTT submission, establish record, iterate

## 13. Bug Root Cause Analysis

### EMA Decay Selection
The original PR #414 used EMA for 20,000 step runs. At full training:
```
0.9999^20000 = 0.135  (13.5% initial weight influence)  ← acceptable
0.9999^7081  = 0.496  (49.6% initial weight influence)  ← catastrophic
```

The correct decay for our actual step count (~7000 steps):
```
0.999^7081 = 0.00085  (0.085% initial weight influence)  ← effectively zero
```
**Fix: change EMA decay from 0.9999 to 0.999**

### SWA Dead Code
PR #414 contains the comment "EMA is better than SWA alone per PR#401" and the post-training sequence applies EMA unconditionally, overwriting whatever base_model contains. SWA state is accumulated in a dict during training but the apply/average step was never added to the post-training sequence.

Since EMA overwrites anyway, SWA's value is in stabilizing the base_model before EMA reads it. Adding SWA apply before EMA apply should recover 0.002-0.005 bpb.

**Fix: add SWA apply block before EMA apply in post-training sequence**

### Post-Training Time Budget
After wall clock stops at 600s, we need to complete:

| Step | Current Time | Fixed Time |
|---|---|---|
| SWA apply | 0s (missing) | ~1s |
| EMA apply | ~1s | ~1s |
| Magnitude pruning | ~5s | ~5s |
| DIAGNOSTIC eval | ~60s | **0s (removed)** |
| Serialization + compress | ~30s | ~30s |
| dist.barrier() | ~1s | ~1s |
| Deserialize + compile | ~30s | ~30s |
| Sliding window eval (×2) | ~120s | **~60s (×1)** |
| **Total** | **~247s** | **~128s** |

Post-training was taking ~247 seconds but the full eval+compile pipeline needs to fit in reasonable time after the 600s training stop. With fixes, we're at ~128 seconds — well within budget.

## 18. Technical Debt & Future Iterations

### Things That Work But Are Suboptimal
- **GPTQ-lite vs Full GPTQ**: We're leaving ~0.008 bpb on the table. Full GPTQ is a layer-wise least-squares optimization that significantly outperforms outlier clipping. Implementation complexity: high.
- **LeakyReLU(0.5)² with hardcoded alpha=0.5**: Should ablate alpha ∈ {0.1, 0.25, 0.5} to find optimal. PR #593 used 0.5 but didn't ablate.
- **BigramHash at 1024**: Reduced from 2048 for artifact size. At 1024 we may be leaving accuracy on the table. Worth testing 1536 if compression improves.

### Things We Haven't Tried
- **Partial RoPE** (16/64 dims instead of full): Appears in several top submissions, small gain but free
- **LN Scale**: LayerNorm scale adjustment, appears in frontier submissions
- **Adaptive per-layer quantization**: Assign int4/int5/int6 per matrix based on sensitivity analysis. Novel, untried by anyone clearly
- **Huffman coding on quantized weights**: Could beat zstd on structured int6 data
- **12th layer**: Budget analysis showed 11L at 15.3MB leaves 700KB — 12th layer costs ~1.0MB compressed, slightly over budget. Would need to reduce bigram or MLP further.

## 20. Phase 5 — Architectural Upgrades: Gated Attention & Value Residual

### Motivation

The goal was to add two well-motivated architectural improvements that require near-zero additional parameters (< 500 bytes of artifact overhead) and cannot negatively affect training due to their identity initialization.

Both modifications were made to the local copy at:
`records/track_10min_16mb/2026-03-25_XSA6_BigramHash4K_HedgeMixer_1.1078/train_gpt.py`

---

### Modification 1: Gated Attention (`attn_gate`)

**Location:** `CausalSelfAttention.__init__` and `CausalSelfAttention.forward`

**What it does:** Adds a learned per-head scalar multiplier to the attention output, giving the model independent control over how strongly each attention head contributes to the residual stream.

In `__init__`:
```python
# Gated Attention: per-head scalar gate, init=1.0 (identity at start)
self.attn_gate = nn.Parameter(torch.ones(num_heads, dtype=torch.float32))
```

In `forward` (after XSA, before reshape):
```python
# Apply gated attention: scale each head's output
gate = self.attn_gate.to(dtype=y.dtype)[None, None, :, None]
y = y * gate
```

**Initialization:** `attn_gate=1.0` is mathematically identical to the original code. The model learns to scale each head from gradient signal alone. 

**Artifact impact:** 11 layers × 8 heads = 88 FP32 scalars = **352 bytes**.

---

### Modification 2: Value Residual (`lambda_v`)

**Location:** `Block.__init__` and `Block.forward`

**What it does:** Adds a learned scalar per block that injects a fraction of the **initial token embedding** (`x0`) directly into each block's output. This helps the model retain token identity through the full depth of the network.

In `__init__`:
```python
# Value Residual: learned scalar to inject initial embedding into output
self.lambda_v = nn.Parameter(torch.zeros(1, dtype=torch.float32))
```

In `forward` (after MLP, before DTG gate):
```python
# Value Residual: add a learned fraction of the initial embedding
x_out = x_out + self.lambda_v.to(dtype=x_out.dtype) * x0
```

**Initialization:** `lambda_v=0.0` means the modification is initially a no-op. It activates only as training proceeds.

**Artifact impact:** 11 layers × 1 scalar = **44 bytes**.

---

### Modification 3: CONTROL_TENSOR_NAME_PATTERNS Registration

Both `attn_gate` and `lambda_v` must NOT be quantized by GPTQ. If quantized to int5/int6, their precision would degrade and the learned gating signal would be corrupted. The script excludes parameters from GPTQ if their name matches a pattern in `CONTROL_TENSOR_NAME_PATTERNS`:

```python
CONTROL_TENSOR_NAME_PATTERNS = tuple(
    pattern for pattern in os.environ.get(
        "CONTROL_TENSOR_NAME_PATTERNS",
        "attn_scale,attn_scales,mlp_scale,mlp_scales,resid_mix,resid_mixes,"
        "q_gain,skip_weight,skip_weights,smear,dtg_gate,ve_layer_scales,"
        "ve_shared.scale,attn_gate,lambda_v",  # <- attn_gate and lambda_v ADDED
    ).split(",") if pattern
)
```

### Root Cause of Improvement

3. Both initialize as **no-ops**, requiring zero hyperparameter tuning and introducing no training instability.
